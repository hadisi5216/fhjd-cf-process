# 技术方案

## 1. 架构概览

系统采用前后端分离的单体应用架构，适合厂区内网部署和后续维护。

```text
固定扫码枪 ──HTTP POST──┐
                        │
管理员浏览器 ──HTTP─────┼── Nginx ── NestJS API ── PostgreSQL
                        │                 ├── 图纸文件存储
公开大屏 ──HTTP + SSE───┘                 └── 定时预警任务
```

生产环境只暴露 Nginx 的 HTTP 端口。Nginx 提供前端静态文件，将 `/api/` 代理到后端，并对 SSE 接口关闭代理缓冲。PostgreSQL、Redis 和后端端口均只在 Docker 网络内使用。Redis 当前作为后续扩展预留，现有核心业务不依赖 Redis。

## 2. 硬件与网络

### 2.1 固定扫码枪

每个工序只配置一台可独立联网的固定扫码枪，现场不配置电脑。设备需要支持二维码识读、网口通信和 HTTP POST 上报，建议使用标准 IEEE 802.3af/802.3at PoE 供电，并确认设备能够配置服务器 IP、端口、请求路径和纯文本上报内容。

扫码枪应设置固定 IP 或由 DHCP 保留地址，便于管理和故障排查；实际工序识别仍优先使用 HTTP URL 中的扫码枪编号。车间存在粉尘、反光或标签污损时，应在安装前验证扫码距离、识读角度和防护等级。

### 2.2 网络与大屏

- 每个扫码点位使用一根网线接入 PoE 交换机，同时传输数据和供电。
- 服务器、扫码枪和电视终端应位于可互通的厂区内网，服务器使用固定 IP。
- 交换机端口和 PoE 功率应保留扩展余量。
- 建议通过 VLAN 或防火墙限制扫码接口只允许厂区网段访问。
- 大屏使用电视、工控机或普通电脑浏览器，推荐 16:9 分辨率和 kiosk 模式。

## 3. 技术选型

| 层级 | 技术 | 用途 |
| --- | --- | --- |
| 前端 | React 19 + TypeScript + Vite | 管理后台与公开大屏 |

| 部署 | Docker Compose + Nginx | 内网一键部署和反向代理 |

## 4. 前端设计

### 4.1 页面与权限

| 路由 | 页面 | 权限 |
| --- | --- | --- |
| `/login` | 管理员登录 | 公开 |
| `/screen` | 产品加工流程实时看板 | 公开只读 |
| `/dashboard` | 后台首页 | 管理员 |
| `/products` | 产品管理 | 管理员 |
| `/products/:id` | 产品详情与流转记录 | 管理员 |
| `/processes` | 工序管理 | 管理员 |
| `/scanners` | 扫码枪管理 | 管理员 |
| `/warnings` | 预警中心 | 管理员 |
| `/settings` | 密码和大屏设置 | 管理员 |

后台统一使用 `ProtectedRoute` 校验本地 JWT。Axios 请求拦截器统一附加 `Authorization: Bearer <token>`，受保护接口返回 `401` 时清除 token、跳转登录页并保留原访问地址。JWT 当前有效期为 8 小时。

### 4.2 数据刷新

- 管理页面由 TanStack Query 管理请求状态和缓存。
- 大屏汇总每 5 秒轮询一次，作为稳定兜底。
- 大屏同时连接 `/api/dashboard/events`。后端完成非重复扫码流转后发送 `dashboard-update` 事件，前端收到后立即使汇总查询失效并重新请求。
- SSE 每 15 秒发送心跳，Nginx 对该路径关闭缓冲并将读取超时设置为 1 小时。
- 重复扫码不改变看板数据，因此不发送刷新事件。

### 4.3 大屏实现

大屏主体采用手写 React 结构和 CSS，不使用 Ant Design 布局组件；时间和刷新图标可使用 Ant Design 图标。看板模拟数据位于前端代码中，每个工序 40 条，由数据库系统设置 `screenPreviewDataEnabled` 控制是否启用。

## 6. 扫码枪 HTTP 协议

### 6.1 推荐配置

```http
POST /api/scanners/scan?name=DM HTTP/1.1
Host: 192.168.1.100
Content-Type: text/plain

J/CLL9-12A-101S1.1
```

| 参数 | 配置 |
| --- | --- |
| 协议 | HTTP |
| 方法 | POST |
| 版本 | HTTP/1.1 |
| 服务器 | 部署服务器 IP |
| 端口 | Nginx 对外端口，默认 80 |
| 路径 | `/api/scanners/scan?name=<扫码枪编号>` |
| 请求体 | 仅产品型号文本 |
| Content-Type | 建议 `text/plain` |

后端也兼容 `application/octet-stream`、JSON 请求体和部分扫码枪省略 Content-Type 的情况，但正式设备统一使用纯文本配置。

### 6.2 扫码枪识别

扫码枪标识可通过查询参数 `scannerCode`、`code`、`name` 或 `scanner` 传入。系统优先使用该标识匹配已启用扫码枪的编号或名称；只有 URL 没有标识时，才使用 Nginx 转发的真实来源 IP 匹配后台 IP。

采用 URL 标识可以避免 Docker、NAT 或反向代理导致来源 IP 不稳定。每台扫码枪必须配置不同且稳定的编号路径。

### 6.3 产品识别

请求体经过首尾空白清理后，按 `productModel` 精确匹配产品。当前数据库没有强制产品型号唯一；若存在多个完全相同型号，系统会选择最近创建的产品。因此需要独立追踪的多件产品必须在二维码内容中使用唯一后缀。

### 6.4 流转事务

每次扫码按以下顺序执行：

1. 匹配已启用扫码枪并更新最近上报时间。
2. 解析产品型号并匹配产品。
3. 在 PostgreSQL 事务中获取该产品的 advisory lock。
4. 重新读取产品最新当前工序。
5. 当前工序与扫码枪绑定工序相同则标记重复，否则执行正常流转。
6. 无论是否重复都新增 `flow_records`。
7. 正常流转时更新产品工序、进入时间和状态。
8. 事务提交后发布大屏更新事件。

数据库事务锁保证同一产品同时被不同扫码枪上报时按顺序处理。

## 7. 核心数据模型

| 表 | 主要用途 |
| --- | --- |
| `admin_users` | 管理员账号、密码哈希、启用状态 |
| `products` | 产品资料、产品工艺说明、当前工序、进入时间和状态 |
| `product_drawings` | 产品图纸的原始名称、存储名称、类型、大小和上传时间 |
| `process_steps` | 工序名称、展示顺序、时限和启用状态 |
| `scanners` | 扫码枪编号、名称、IP、绑定工序和最近上报时间 |
| `flow_records` | 每次扫码或管理员调整形成的完整历史 |
| `warnings` | 超时预警、处理状态和处理说明 |
| `system_settings` | 大屏模拟数据等系统配置 |

### 7.1 产品状态

- `PENDING`：待加工，尚未进入工序。
- `IN_PROGRESS`：正在加工。
- `FINISHED`：已经扫描或手动调整到完工。
- `OVERDUE`：保留的超时状态枚举；当前超时判断以开放预警和进入时间为准。

### 7.2 流转记录

`flow_records` 保存产品、工序、扫码内容、扫码枪、时间、来源、重复标记和备注。

- `source = SCAN`：扫码枪上报。
- `source = MANUAL`：管理员手动调整。
- `is_duplicate = true`：扫描工序等于产品事务内最新当前工序。
- 管理员调整允许 `scanner_id` 为空，并使用 `note` 保存说明。

### 7.3 超时任务

后端每 5 分钟扫描一次正在加工的产品。当前时间减去 `current_entered_at` 达到工序 `timeout_hours` 时，为该产品和工序创建一条开放预警；已存在相同开放预警时不重复创建。完工工序时限为 0，不参与扫描。

### 7.4 产品图纸存储

图纸文件本体写入 `UPLOAD_DIR` 指定的磁盘目录，PostgreSQL 只保存文件元数据。单个文件上限 50 MB，允许 PDF、常见图片、DWG/DXF 及常见三维图纸格式。Docker 使用独立命名卷持久化上传目录，容器重建不会删除图纸。

图纸读取接口需要管理员 JWT。前端先以鉴权请求获取 Blob，再用临时对象 URL 在新标签预览 PDF，避免把 token 放在地址栏。非 PDF 文件使用附件下载方式。

### 7.5 产品工艺说明

`products.manufacturing_process` 保存最多 5000 字的多行自由文本。该字段通过产品更新接口单独维护，只用于产品详情展示；更新时不调用流转事务，不修改 `current_process_id`、`current_entered_at` 或产品状态，也不新增 `flow_records` 和预警。

## 9. 安全与可靠性

- 密码使用 bcrypt 哈希保存，不保存明文。
- 生产环境必须更换 `JWT_SECRET` 和数据库默认密码。
- NestJS 全局启用 DTO 白名单验证、CORS 和 Helmet。
- 管理接口使用 JWT Guard；扫码、健康检查和公开大屏接口按用途开放。
- 扫码接口只应暴露在可信厂区内网，建议通过交换机 VLAN、防火墙或访问控制限制来源网段。
- PostgreSQL 数据卷必须定期备份，更新版本前先执行数据库备份。
- 图纸上传目录不保存可执行文件；文件名由服务端随机生成，下载时恢复原始名称。
- 数据库备份不包含图纸文件，必须同时备份 PostgreSQL 数据和上传文件卷。
- 目前实时事件总线为单后端进程内实现。若未来横向部署多个 API 实例，需要改为 Redis Pub/Sub 等跨实例事件通道。
