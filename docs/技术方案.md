# 技术方案：产品加工流程管理系统

## 1. 建设模式

系统采用 B/S 架构，由后台管理系统、大屏看板、扫码枪接入服务和数据库组成。

工序现场不部署电脑。每个工序配置固定式网口扫码枪，扫码枪通过网线接入网络，建议使用 PoE 交换机供电。产品到达某道工序后，扫码枪扫描产品上已有二维码。后台根据扫码枪来源识别工序，根据扫码内容识别产品，并生成流转记录。

二维码由其他地方单独打印并贴到产品上，本系统不负责二维码生成和打印。

## 2. 技术选型

| 层级 | 推荐技术 | 说明 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 用于后台管理页面和大屏看板 |
| UI 组件 | Ant Design | 适合后台表单、表格、弹窗；大屏主体不使用 Ant Design 布局组件 |
| 图表 | ECharts，可选 | 后续统计分析需要时再引入 |
| 后端 | Node.js 20 + NestJS | 适合 REST API、扫码枪 HTTP 接入和模块化后台开发 |
| 登录 | JWT + NestJS Guards | 当前只做管理员登录 |

管理员 JWT 有效期为 8 小时。前端路由加载时校验 token 的 `exp`，接口运行期间收到 `401` 时由 Axios 响应拦截器统一清除失效 token 并跳转登录页，重新登录后返回失效前页面。公开大屏接口不依赖管理员 token。
| ORM | Prisma | 适合 PostgreSQL 建模、查询和迁移 |
| 数据库 | PostgreSQL | 存储产品、工序、扫码记录、预警记录 |
| 缓存 | Redis，可选 | 用于看板缓存和热点统计 |
| 定时任务 | @nestjs/schedule | 用于 72 小时超时预警扫描 |
| 部署 | Docker + Docker Compose | 适合厂区内网服务器部署、备份和迁移 |

当前项目按 **React + Node.js/NestJS + PostgreSQL + Docker** 实施。

## 3. 前端设计

前端分为两个入口：

- 后台管理页面：管理员登录后维护产品、工序、扫码枪和预警。
- 大屏看板页面：公开只读展示产品状态、各工序数量和超时预警。

当前大屏看板实现口径：

- `/screen` 不接入后台布局和权限校验，作为独立公开只读页面。
- 看板主体使用 React 手写 DOM 和 CSS 实现，不使用 Ant Design 布局组件承载大屏主体。
- 时间和刷新图标使用 Ant Design 图标。
- 顶部显示产品总数、加工中、已完工、已超时四个指标，四个指标卡以屏幕整体居中。
- 主体按打磨、装配、喷漆、包覆展示当前在制产品。
- 工序内部使用原生表格展示产品型号和当前工序已用时。
- 工序列表隐藏滚动条，但保留内部滚动能力。

| 路由 | 页面 | 登录要求 |
| --- | --- | --- |
| `/login` | 管理员登录 | 不需要 |
| `/dashboard` | 后台首页 | 需要 |
| `/products` | 产品管理 | 需要 |
| `/processes` | 工序管理 | 需要 |
| `/scanners` | 扫码枪管理 | 需要 |
| `/warnings` | 预警中心 | 需要 |
| `/screen` | 大屏看板 | 不需要 |

当前阶段取消任务单、独立资产台账、二维码打印页面、报表分析页面。产品相关维护统一进入产品管理。

## 4. 后端设计

后端采用 NestJS 提供 REST API 和扫码枪接入接口。

核心职责：

- 管理员登录。
- 管理员账号密码维护。
- 产品管理。
- 工序管理。
- 扫码枪绑定管理。
- 接收扫码枪上报数据。
- 生成工序流转记录。
- 计算超时预警。
- 提供后台首页和大屏看板统计接口。

权限策略：

- 后台管理接口需要管理员登录。
- 当前阶段没有工序负责人登录。
- 大屏看板读取接口可开放为内网只读接口，页面不包含编辑、删除、配置等管理操作。

## 5. 二维码与扫码内容

本系统不生成二维码、不打印二维码、不维护二维码标签模板。

二维码由其他系统或其他环节生成并贴到产品上。本系统只接收扫码枪上传的扫码内容，并根据内容匹配产品。

当前扫码内容优先采用产品型号。

示例：

```text
J/CL19-12A-101S1.1
```

若后续出现同一产品型号对应多件产品同时流转，建议改为以下任一方式：

- 产品型号 + 流水号
- 系统内部产品编号
- 外部二维码唯一编号

## 6. 扫码枪接入

扫码枪当前按 HTTP POST 方式接入，统一上报到后台接口 `/api/scanners/scan`。

扫码枪发送扫码得到的产品型号，并建议在 HTTP 请求路径中携带扫码枪标识，例如 `/api/scanners/scan?name=DM`。后台优先根据 URL 参数匹配扫码枪编号或扫码枪名称，再根据扫码枪绑定关系判断工序；来源 IP 仅作为兜底匹配方式。

HTTP 上报是短连接模式，不存在持续连接在线。扫码枪管理不展示在线或离线状态，只保留最近上报时间，表示该扫码枪最近一次发送扫码数据的时间。

扫码流程：

1. 扫码枪扫描产品二维码。
2. 扫码枪向服务器 HTTP 接口发送产品型号文本。
3. 后台根据 URL 参数中的扫码枪标识识别工序。
4. 后台根据扫码内容匹配产品。
5. 后台生成产品进入该工序的流转记录。

HTTP 上报接口：

```http
POST /api/scanners/scan?name=DM
Content-Type: text/plain
```

支持的扫码枪标识参数：

- `scannerCode`：推荐使用，对应后台“扫码枪编号”。
- `code`：等同于 `scannerCode`。
- `name`：兼容部分扫码枪配置习惯，可匹配后台“扫码枪编号”或“扫码枪名称”。
- `scanner`：兼容参数名。

上报内容示例：

```text
J/CLL9-12A-101S1.1
```

接口也兼容 JSON 格式：

```json
{
  "content": "J/CLL9-12A-101S1.1"
}
```

## 7. 核心数据表

- `admin_user`：管理员账号。
- `product`：产品档案。
- `process_step`：工序配置。
- `scanner`：扫码枪档案及绑定工序。
- `flow_record`：产品流转记录。用于保存该产品已经经过的全部工序历史，包含产品、工序、进入时间、来源、扫码枪、操作人和说明。
- `warning`：超时预警记录。

`flow_record` 关键字段：

- `product_id`：产品。
- `process_step_id`：进入的工序。
- `scanned_at`：进入工序时间。
- `source`：记录来源，`SCAN` 表示扫码进入，`MANUAL` 表示管理员调整。
- `scanner_id`：扫码枪，管理员调整记录允许为空。
- `scan_content`：扫码内容，当前优先为产品型号。
- `note`：说明，例如“管理员手动调整工序”。

## 9. 流转规则

- 扫码枪绑定一个工序。
- HTTP 扫码内容当前按产品型号匹配产品。
- 后台优先根据 URL 参数中的扫码枪标识找到扫码枪，再根据扫码枪绑定关系找到工序；来源 IP 只作为兜底。
- 扫码成功后，产品当前工序更新为扫码枪绑定工序。
- 产品首次扫描进入当前工序时，当前工序进入时间更新为本次扫码时间。
- 同一产品首次扫描某工序时，以该次扫码时间记录进入时间；再次扫描同一工序时新增 `is_duplicate = true` 的重复扫码记录。
- 重复扫码仅留痕，不更新产品当前工序、当前工序进入时间和状态，也不会造成流程倒退。
- 上一工序滞留时间自然结束。
- 产品详情中的流转记录按进入时间展示该产品已经经过的全部工序。
- 当产品真实进入某工序但扫码枪异常、漏扫或误扫时，管理员可手动调整工序。
- 手动调整工序会更新产品当前工序和当前工序进入时间，并生成来源为 `MANUAL` 的“管理员调整”流转记录。
- 同一产品同一工序的每次重复扫码均保留记录，并在产品详情中标记“重复扫码”。
- 产品超过 72 小时没有进入下一工序时，生成超时预警。

## 10. 验收重点

- 管理员能登录后台并修改密码。
- 管理员能手动新增、编辑、删除产品。
- 管理员能维护工序和扫码枪。
- 固定扫码枪能上传扫码内容。
- 后台能根据扫码枪来源识别工序。
- 后台能根据扫码内容识别产品。
- 扫码后产品当前工序立即更新。
- 产品详情能以列表展示全部历史流转记录，包括扫码进入和管理员调整。
- 管理员能在扫码枪异常、漏扫或误扫时手动调整工序，并留下管理员调整记录。
- 看板能显示当前工序、进入时间、滞留时长。
- 超过 3 天未进入下一工序时能自动预警。
- 大屏看板无需登录即可只读访问。
