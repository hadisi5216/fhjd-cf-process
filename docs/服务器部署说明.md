# 服务器部署说明

本文适用于已安装 Docker 和 Docker Compose 的 Linux 或 Windows 服务器。生产环境建议使用固定内网 IP，并确保扫码枪能够访问该 IP 的 HTTP 端口。

## 1. 部署前检查

```bash
docker version
docker compose version
```

服务器至少需要开放：

- `80/tcp`：浏览器、公开大屏和扫码枪统一入口。
- Docker 拉取镜像所需的外网或内网镜像仓库访问。

后端 `3000`、PostgreSQL 和 Redis 不需要对宿主机开放。

## 2. 准备项目

Linux 推荐目录：

```bash
mkdir -p /opt/fhjd-cf-process
cd /opt/fhjd-cf-process
```

Windows 可放在固定目录，例如：

```powershell
Set-Location C:\server\fhjd-cf-process
```

项目目录至少应包含 `apps`、`packages`、`deploy`、`package.json`、`package-lock.json` 和 `.env.example`。

复制环境变量：

```powershell
Copy-Item .env.example .env
```

Linux 使用：

```bash
cp .env.example .env
```

## 3. 生产环境配置

编辑项目根目录 `.env`：

```env
POSTGRES_DB=fhjd_cf_process
POSTGRES_USER=fhjd_cf
POSTGRES_PASSWORD=请替换为强密码
DATABASE_URL=postgresql://fhjd_cf:请替换为强密码@postgres:5432/fhjd_cf_process?schema=public
JWT_SECRET=请替换为足够长的随机字符串
WEB_PORT=80
UPLOAD_DIR=/app/storage/uploads
```

注意：

- `POSTGRES_PASSWORD` 和 `DATABASE_URL` 中的密码必须一致。
- 密码包含 `@`、`:`、`/`、`#` 等字符时，`DATABASE_URL` 中必须进行 URL 编码。
- `JWT_SECRET` 不得使用模板默认值。
- `WEB_PORT=80` 时浏览器直接输入服务器 IP，不需要写端口。
- `UPLOAD_DIR` 必须与 Compose 中的上传附件持久化卷挂载目录一致，通常保持默认值。

镜像下载不稳定时，可在 `.env` 中替换镜像地址或 npm 源：

```env
POSTGRES_IMAGE=postgres:15-alpine
REDIS_IMAGE=redis:7-alpine
NODE_IMAGE=node:20-alpine
NGINX_IMAGE=nginx:1.27-alpine
NPM_REGISTRY=https://registry.npmmirror.com/
```

## 4. 首次启动

在项目根目录执行：

```bash
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml up -d --build
```

确认容器状态：

```bash
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml ps
```

容器名称应为：

```text
fhjd-cf-process-web
fhjd-cf-process-api
fhjd-cf-process-postgres
fhjd-cf-process-redis
```

同步数据库结构并写入初始数据：

```bash
docker exec fhjd-cf-process-api npx prisma db push
docker exec fhjd-cf-process-api npm run prisma:seed
```

### 4.1 数据库命令说明

| 命令 | 何时执行 | 作用 |
| --- | --- | --- |
| `docker exec fhjd-cf-process-api npx prisma db push` | 首次部署，或代码新增、删除、修改数据库表和字段后 | 将 `schema.prisma` 中的表结构同步到 PostgreSQL |
| `docker exec fhjd-cf-process-api npm run prisma:seed` | 首次部署，或默认管理员、工序、扫码枪、系统设置等基础数据发生变化后 | 写入或补齐系统基础数据 |

两条命令都需要执行时，必须先执行 `prisma db push`，再执行 `prisma:seed`。`prisma:seed` 只负责写入基础数据，不能创建缺失的数据表；如果只修改普通前后端代码，则两条命令都不需要执行。

seed 默认创建：

- 管理员 `admin`，新库初始密码 `admin123456`。
- 工序：打磨、装配、喷漆、包覆、完工。
- 扫码枪：`DM`、`ZP`、`PQ`、`BF`、`WG`。
- 大屏模拟数据开关：开启。

首次登录后立即修改管理员密码，并在“系统设置”中关闭大屏模拟数据。重复执行 seed 不会重置已经修改的管理员密码。

### 4.2 产品上传附件存储位置

产品图纸和工艺流程附件本体不写入 PostgreSQL，也不保存在代码目录中。Docker 部署时的存储关系如下：

| 内容 | 存储位置 |
| --- | --- |
| 图纸、工艺流程附件本体 | API 容器内 `/app/storage/uploads` |
| 持久化存储 | Docker 命名卷 `fhjd-cf-process_uploads_data` |
| 图纸元数据 | PostgreSQL 表 `product_drawings` |
| 工艺流程附件元数据 | PostgreSQL 表 `product_process_attachments` |

服务端会使用随机文件名保存附件，页面展示和下载时仍使用上传时的原始文件名。重建或替换 API 容器不会删除命名卷中的附件；执行 `docker compose down -v` 会删除该卷，生产环境禁止使用。

查看当前上传文件：

```bash
docker exec fhjd-cf-process-api sh -lc 'find "$UPLOAD_DIR" -maxdepth 1 -type f -exec ls -lh {} \;'
```

查看命名卷及 Docker 管理的实际挂载位置：

```bash
docker volume inspect fhjd-cf-process_uploads_data
```

将全部上传附件导出到当前目录的 `product-files-export` 文件夹：

```bash
docker cp fhjd-cf-process-api:/app/storage/uploads ./product-files-export
```

磁盘中的附件使用随机文件名保存；需要按原始中文名称获取单个文件时，应在产品详情页中预览或下载。

Windows Docker Desktop 返回的 `/var/lib/docker/volumes/...` 位于 Docker 的 Linux 虚拟机内，不是普通 Windows 目录，不应直接进入或手工修改。附件导出和备份应使用本文第 9 节的 Docker 卷备份命令。

## 5. 访问与健康检查

假设服务器 IP 是 `192.168.1.100`：

```text
后台入口：http://192.168.1.100/
公开大屏：http://192.168.1.100/screen
健康检查：http://192.168.1.100/api/health
```

后台需要管理员登录，`/screen` 无需登录。网页会尝试进入全屏，但浏览器可能要求首次点击或按键。无人值守电视应把浏览器配置为 kiosk 模式，并将 `/screen` 设置为启动页。

## 6. 扫码枪 HTTP 配置

所有扫码枪使用相同服务器地址和端口，不同工序只修改路径中的编号。

| 配置项 | 值 |
| --- | --- |
| 协议 | HTTP |
| 请求方式 | POST |
| HTTP 版本 | 1.1 |
| 服务器地址 | 部署服务器内网 IP，例如 `192.168.1.100` |
| 服务器端口 | `80` |
| 请求头 | `Content-Type: text/plain` |
| 上报内容 | 只发送扫码得到的产品型号 |

路径对应关系：

| 工序 | 路径 |
| --- | --- |
| 打磨 | `/api/scanners/scan?name=DM` |
| 装配 | `/api/scanners/scan?name=ZP` |
| 喷漆 | `/api/scanners/scan?name=PQ` |
| 包覆 | `/api/scanners/scan?name=BF` |
| 完工 | `/api/scanners/scan?name=WG` |

URL 编号会优先匹配后台扫码枪编号或名称。只有 URL 不带编号时才使用来源 IP 兜底，因此 Docker 部署不影响正常识别。HTTP 是短连接，后台“最近上报时间”表示最后一次收到数据的时间，不代表持续在线状态。

## 7. 更新版本

更新前先备份数据库，然后上传或拉取新代码：

```bash
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml up -d --build
docker exec fhjd-cf-process-api npx prisma db push
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml ps
```

附件功能首次上线时，`prisma db push` 会创建 `product_drawings`、`product_process_attachments` 表，Compose 会自动创建 `uploads_data` 命名卷。

如果本次版本修改了默认工序、扫码枪或系统设置，再执行：

```bash
docker exec fhjd-cf-process-api npm run prisma:seed
```

项目引入正式 Prisma migrations 后，生产环境将 `db push` 替换为：

```bash
docker exec fhjd-cf-process-api npx prisma migrate deploy
```

## 8. 日常运维

```bash
# 状态
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml ps

# 全部日志
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml logs -f

# 后端日志
docker logs -f fhjd-cf-process-api

# 重启
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml restart

# 停止
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml down
```

不要在生产环境执行 `down -v`，该命令会同时删除 PostgreSQL 数据卷和上传附件文件卷。

## 9. 数据与上传附件备份

Linux 备份：

```bash
docker exec fhjd-cf-process-postgres sh -c 'pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB"' > fhjd-cf-process-$(date +%F).sql
```

Linux 恢复前应先确认备份文件和目标库：

```bash
cat backup.sql | docker exec -i fhjd-cf-process-postgres sh -c 'psql -U "$POSTGRES_USER" "$POSTGRES_DB"'
```

Windows 使用模板默认数据库用户和库名时，也可运行项目脚本：

```powershell
.\scripts\backup-db.ps1 -Output .\backup-fhjd-cf-process.sql
.\scripts\restore-db.ps1 -InputFile .\backup-fhjd-cf-process.sql
```

生产环境修改了数据库用户或库名时，应同步修改脚本参数或使用容器内环境变量命令。

数据库 SQL 备份不包含产品图纸和工艺流程附件本体。还需要单独备份上传卷：

```bash
docker run --rm -v fhjd-cf-process_uploads_data:/data -v "$PWD":/backup alpine tar czf /backup/fhjd-cf-process-uploads.tar.gz -C /data .
```

恢复上传附件前先停止 API 容器，再将压缩包解压回同一个 `uploads_data` 卷。备份文件应与同一时间点的数据库备份一起保存。

## 10. 常见问题

### 10.1 80 端口被占用

错误中包含 `bind: Only one usage of each socket address` 或 `ports are not available`，表示 IIS、Nginx、其他程序或旧容器已占用端口。

Windows 检查：

```powershell
Get-NetTCPConnection -LocalPort 80 -State Listen
docker ps --format "table {{.Names}}\t{{.Ports}}"
```

停止占用服务，或把 `.env` 中 `WEB_PORT` 改为未占用端口。后端容器不需要映射宿主机 `3000` 端口。

### 10.2 前端能打开但接口失败

```bash
docker compose --env-file .env -p fhjd-cf-process -f deploy/docker-compose.yml ps
docker logs --tail 200 fhjd-cf-process-api
docker logs --tail 100 fhjd-cf-process-postgres
```

重点检查 API 是否反复重启、`DATABASE_URL` 密码是否一致，以及是否已执行 `prisma db push`。

### 10.3 提示数据表不存在

```bash
docker exec fhjd-cf-process-api npx prisma db push
docker exec fhjd-cf-process-api npm run prisma:seed
```

如果 seed 报 TypeScript 配置错误，通常是 API 镜像仍为旧版本，重新执行带 `--build` 的启动命令后再运行。

### 10.4 seed 提示脚本不存在

正确命令是：

```bash
docker exec fhjd-cf-process-api npm run prisma:seed
```

确认没有把 `seed` 写成 `see`，并确认容器是当前项目构建的 `fhjd-cf-process-api`。

### 10.5 扫码枪有上报但产品不流转

1. 查看 `docker logs -f fhjd-cf-process-api`。
2. 确认请求路径编号与后台已启用扫码枪一致。
3. 确认请求体只有产品型号，没有额外前后缀或不可见字符。
4. 确认产品管理中存在完全一致的产品型号。
5. 检查扫码枪“最近上报时间”。时间更新但未流转通常表示产品未匹配。

### 10.6 看板没有立即刷新

浏览器应连接 `/api/dashboard/events`。检查 Nginx 配置是否为项目当前版本，并确认该接口响应类型为 `text/event-stream`。即使 SSE 断开，页面仍会每 5 秒重新拉取一次汇总。
